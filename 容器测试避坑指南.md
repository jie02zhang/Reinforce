# ⚠️ 容器测试避坑指南

## 概述

本文档详细说明在 Docker 容器中测试安全加固脚本的特殊限制、潜在风险，以及如何安全地过渡到物理机/虚拟机测试。

---

## 1. 容器与物理机/虚拟机的核心差异

### 1.1 内核共享

**问题**: 容器与宿主机共享内核，容器内的内核参数修改会影响宿主机或其他容器。

**影响**:
- `sysctl -w net.*` 某些参数在容器内无法修改
- 修改 `kernel.*` 参数可能影响宿主机稳定性

**解决方案**:
```bash
# 在容器内检测是否为容器环境
if [ -f /.dockerenv ] || grep -qa docker /proc/1/cgroup 2>/dev/null; then
    log "[WARN] 容器环境检测到，跳过某些内核参数修改"
    # 仅修改允许在容器内修改的参数
fi
```

**允许在容器内修改的参数** (需要 `--sysctl` 或 `--privileged`):
- `net.ipv4.ip_local_port_range`
- `net.ipv4.tcp_*` (部分)
- `net.core.somaxconn`

**禁止在容器内修改的参数**:
- `kernel.randomize_va_space`
- `kernel.sysrq`
- `vm.swappiness`

---

### 1.2 INIT 进程差异

**问题**: 容器内的 PID 1 通常不是完整的 init 系统（如 systemd），导致服务管理命令行为异常。

**影响**:
- `systemctl` 命令可能不可用或报错
- 服务启动/停止/重启可能失败
- `systemctl enable` 在容器内无效

**检测与适配**:
```bash
# 检测是否使用 systemd
if command -v systemctl &>/dev/null && systemctl --version &>/dev/null; then
    IS_SYSTEMD=true
else
    IS_SYSTEMD=false
    log "[WARN] 非 systemd 环境，服务管理将使用备选方案"
fi

# 服务管理适配
svc_ctrl() {
    local op="$1"
    local name="$2"
    
    if $IS_SYSTEMD; then
        systemctl "$op" "${name}.service" 2>/dev/null || true
    elif command -v service &>/dev/null; then
        service "$name" "$op" >/dev/null 2>&1 || true
    else
        # 直接调用二进制（适用于容器）
        case "$op" in
            start) "/usr/sbin/${name}" >/dev/null 2>&1 & ;;
            stop) pkill -f "$name" 2>/dev/null || true ;;
            restart) pkill -f "$name" 2>/dev/null; "/usr/sbin/${name}" >/dev/null 2>&1 & ;;
        esac
    fi
}
```

---

### 1.3 安全模块限制

**问题**: 容器可能禁用或限制 SELinux、AppArmor。

**影响**:
- SELinux 策略无法在容器内修改
- AppArmor profile 可能无法加载

**检测**:
```bash
# 检测 SELinux
if command -v getenforce &>/dev/null; then
    SELINUX_STATUS=$(getenforce 2>/dev/null || echo "Disabled")
    log "[INFO] SELinux 状态: $SELINUX_STATUS"
else
    log "[INFO] SELinux 未安装"
fi

# 检测 AppArmor
if [ -d /etc/apparmor.d ]; then
    log "[INFO] AppArmor 已安装"
    # 注意：容器内可能无法修改 AppArmor profile
fi
```

## 2. Docker 容器测试配置

### 2.1 推荐的 Docker 运行配置

```bash
# 运行容器时进行充分授权（仅测试环境）
docker run -d \
  --name security-test \
  --privileged \                           # 授予所有权限（仅测试用）
  --cap-add=ALL \                          # 添加所有能力
  --security-opt seccomp=unconfined \     # 禁用 seccomp 限制
  --security-opt apparmor=unconfined \    # 禁用 AppArmor
  -v $(pwd)/security_hardening.sh:/tmp/security_hardening.sh:ro \
  -v $(pwd)/test_results:/test_results \
  ubuntu:22.04 \
  /sbin/init

# 或者使用 docker-compose (已提供)
docker-compose up -d
```

### 2.2 容器内初始化 systemd

**问题**: 默认 Ubuntu/Debian 容器没有 systemd 作为 PID 1。

**解决方案**:
```dockerfile
# Dockerfile 示例
FROM ubuntu:22.04

# 安装 systemd
RUN apt-get update && \
    apt-get install -y systemd openssh-server sudo cron && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/* /tmp/* /var/tmp/*

# 移除不必要的 systemd 单位
RUN (cd /lib/systemd/system/sysinit.target.wants/ && for i in *; do [ "$i" == "systemd-tmpfiles-setup.service" ] || rm -f "$i"; done) && \
    rm -f /lib/systemd/system/multi-user.target.wants/* && \
    rm -f /etc/systemd/system/*.wants/* && \
    rm -f /lib/systemd/system/local-fs.target.wants/* && \
    rm -f /lib/systemd/system/sockets.target.wants/*udev* && \
    rm -f /lib/systemd/system/sockets.target.wants/*initctl* && \
    rm -f /lib/systemd/system/basic.target.wants/*

VOLUME [ "/sys/fs/cgroup" ]

CMD ["/sbin/init"]
```

构建并运行:
```bash
docker build -t ubuntu-systemd .
docker run -d --name test-ubuntu --privileged -v /sys/fs/cgroup:/sys/fs/cgroup:rw ubuntu-systemd
```

## 3. 容器内无法充分测试的项目

### 3.1 内核参数 (sysctl)

**无法完全测试**: 部分内核参数需要宿主机级别权限。

**建议**:
- 在容器内测试配置文件的修改逻辑（`/etc/sysctl.conf`）
- 使用 `--privileged` 或 `--sysctl` 测试部分参数
- **必须在物理机/虚拟机上最终验证**

### 3.2 SELinux/AppArmor

**无法完全测试**: 容器内的安全模块可能被禁用或限制。

**建议**:
- 在容器内测试 PAM、文件权限等不依赖安全模块的加固项
- 在启用了 SELinux/AppArmor 的虚拟机上单独测试

### 3.3 引导加载程序 (GRUB)

**无法测试**: 容器没有 GRUB。

**建议**:
- 在物理机/虚拟机上手动验证 `/etc/default/grub` 配置
- 测试 `grub-mkconfig` 命令是否成功

### 3.4 磁盘加密与 LUKS

**无法测试**: 容器不使用物理磁盘。

**建议**:
- 此项目需要专门的测试环境（加密磁盘的虚拟机）

### 3.5 物理控制台安全

**无法测试**: 容器没有物理控制台。

**建议**:
- 在物理机/虚拟机上测试 `/etc/securetty`、`/etc/issue` 等

## 4. 测试策略：从容器到生产

### 4.1 阶段 1: 容器测试 (快速验证)

**目标**: 验证脚本基本逻辑、文件操作、包管理。

**测试内容**:
- ✅ 文件权限修改
- ✅ 配置文件编辑 (sed/awk)
- ✅ 包安装 (apt/yum/zypper)
- ✅ 用户/组管理
- ✅ 基础 PAM 配置
- ⚠️ 部分 systemd 服务操作 (需要 --privileged)

**不测试**:
- ❌ 内核参数 (sysctl)
- ❌ SELinux/AppArmor
- ❌ 防火墙规则 (需要特殊配置)
- ❌ 引导加载程序

**执行**:
```bash
# 在容器内运行
docker exec -it security-test-ubuntu bash
bash /tmp/security_hardening.sh --dry-run   # 先干跑
bash /tmp/security_hardening.sh              # 实际执行
bash /tmp/health_check.sh post              # 验证
```

---

### 4.2 阶段 2: 虚拟机测试 (中等保真)

**目标**: 验证系统级加固项。

**推荐环境**:
- VirtualBox / VMware / KVM 虚拟机
- 启用 systemd
- 启用 SELinux (RHEL) 或 AppArmor (Ubuntu)

**测试内容**:
- ✅ 所有容器测试项目
- ✅ 内核参数 (sysctl)
- ✅ 防火墙 (firewalld/ufw)
- ✅ SELinux/AppArmor 基础配置
- ✅ 服务管理 (systemctl)
- ⚠️ 部分引导加载程序配置

**执行**:
```bash
# 在虚拟机内运行
scp security_hardening.sh user@vm-ip:/tmp/
ssh user@vm-ip
sudo bash /tmp/security_hardening.sh --dry-run
sudo bash /tmp/security_hardening.sh
```

---

### 4.3 阶段 3: 物理机测试 (生产保真)

**目标**: 最终验证，确保生产环境安全。

**测试内容**:
- ✅ 所有虚拟机的测试项目
- ✅ 引导加载程序 (GRUB) 安全
- ✅ 磁盘加密 (如果需要)
- ✅ 物理控制台安全
- ✅ 硬件相关的内核参数

**重要**: 在物理机上测试时，务必：
1. 准备应急启动盘 (Live CD/USB)
2. 备份重要数据
3. 先在测试物理机上验证
4. 准备回滚计划

## 5. 常见坑点与解决方案

### 5.1 SSH 被锁问题

**问题**: 加固脚本修改 SSH 配置后，可能无法远程登录。

**预防**:
```bash
# 在脚本中增加预检查
pre_check_ssh() {
    log "[PRE-CHECK] 备份 SSH 配置"
    backup_file /etc/ssh/sshd_config
    
    # 确保至少有一个允许的认证方式
    if ! grep -q "^PasswordAuthentication yes" /etc/ssh/sshd_config && \
       ! grep -q "^PubkeyAuthentication yes" /etc/ssh/sshd_config; then
        log "[WARN] SSH 可能禁用所有认证方式，自动允许密码认证"
        echo "PasswordAuthentication yes" >> /etc/ssh/sshd_config
    fi
    
    # 验证配置
    if ! sshd -t 2>&1; then
        log "[ERROR] SSH 配置验证失败，中止修改"
        return 1
    fi
}
```

**紧急恢复**:
```bash
# 通过控制台或带外管理登录
cp /etc/ssh/sshd_config.bak /etc/ssh/sshd_config
systemctl restart sshd
```

---

### 5.2 防火墙锁定问题

**问题**: 防火墙规则配置错误，导致无法远程访问。

**预防**:
```bash
# 在脚本中增加防火墙规则验证
pre_check_firewall() {
    if command -v firewall-cmd &>/dev/null; then
        # 先添加临时规则（重启后失效）
        firewall-cmd --add-service=ssh
        firewall-cmd --reload
        
        # 等待用户确认
        echo "防火墙规则已更新，请在新终端测试连接"
        echo "如果连接正常，运行: firewall-cmd --runtime-to-permanent"
        echo "如果连接异常，运行: firewall-cmd --reload"
        sleep 10
    fi
}
```

---

### 5.3 包管理器损坏

**问题**: 错误地修改仓库配置，导致无法安装软件包。

**预防**:
```bash
# 备份仓库配置
backup_repo_config() {
    local repo_dir=""
    case "$PKG_MGR" in
        apt) repo_dir="/etc/apt" ;;
        yum|dnf) repo_dir="/etc/yum.repos.d" ;;
        zypper) repo_dir="/etc/zypp/repos.d" ;;
    esac
    
    if [ -d "$repo_dir" ]; then
        backup_file "$repo_dir"
        log "[BKUP] 已备份仓库配置: $repo_dir"
    fi
}
```

## 6. 过渡到生产环境的检查清单

在将脚本部署到生产环境前，完成以下检查：

### 6.1 代码审查
- [ ] 所有变量都用双引号包裹
- [ ] 通过 ShellCheck 检查 (`shellcheck security_hardening.sh`)
- [ ] 没有硬编码的密码或密钥
- [ ] 所有外部输入都经过验证

### 6.2 功能测试
- [ ] 在容器中测试通过
- [ ] 在虚拟机中测试通过
- [ ]  Dry-run 模式输出合理
- [ ] 回滚功能正常

### 6.3 兼容性验证
- [ ] 在目标发行版的所有版本上测试
- [ ] 检查包名差异 (如 libpam-pwquality vs pam_pwquality)
- [ ] 检查服务名差异 (如 sshd vs ssh)
- [ ] 检查配置文件路径差异

### 6.4 安全审查
- [ ] 不会意外锁定管理员
- [ ] 不会删除关键系统文件
- [ ] 不会禁用关键系统服务
- [ ] 备份功能正常

### 6.5 文档与回滚
- [ ] 用户文档完整
- [ ] 回滚步骤明确
- [ ] 应急联系方式可用

## 8. 总结

1. **容器测试** 适合快速验证基本逻辑，但无法测试系统级加固项
2. **虚拟机测试** 是性价比最高的测试方式，可以验证大部分生产场景
3. **物理机测试** 是生产部署前的必要步骤，尤其是涉及引导加载程序、磁盘加密等项目
4. **渐进式测试**: 容器 → 虚拟机 → 物理机 → 生产
5. **永远准备好回滚计划**

---

**最后提醒**: 即使脚本在容器、虚拟机、测试物理机上测试通过，生产环境仍可能存在差异。首次在生产环境部署时，务必：
- 在维护窗口进行
- 准备应急方案
- 先在小范围（1-2 台）试点
- 逐步推广
